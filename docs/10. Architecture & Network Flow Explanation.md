# Architecture and Network Flow Documentation

## 1. System Overview

This project implements a high-performance, distributed web infrastructure designed for scalability and resilience. The architecture follows a multi-tier approach, separating the entry point, application processing, asynchronous messaging, and persistent storage layers.

The flow of data through the network is designed for efficiency and security, ensuring that internal services are never exposed to the public internet.

### Step 2: Application Processing (Web Tier)

* 
**Action:** The Load Balancer forwards the request to a **LiteSpeed Web Server**.
**Action:** Instead of making the user wait for heavy data processing (like analytics or logs), the web server produces a message and sends it to **[Kafka](04-kafka-broker.md)**.

### Step 3: Asynchronous Messaging (Messaging Tier)

**Logic:** The final processed data is written into the **[Primary Database](07-database-server.md)**.
**Action:** Instead of making the user wait for heavy data processing (like analytics or logs), the web server produces a message.


* 
**Flow:** Data is sent to the **Kafka Broker** on Port 9092.


* **Logic:** Kafka acknowledges the message and stores it in a topic. This "decouples" the web server from the storage layer, ensuring that a database slowdown doesn't crash the website.



### Step 4: Data Persistence (Storage Tier)

* 
**Action:** **Kafka Consumers** (Database Writers) monitor the Kafka topics.


* 
**Flow:** The consumer pulls the data from Kafka and performs necessary transformations.


* 
**Logic:** The final processed data is written into the **Primary Database**.

## 4. Security and Isolation

To satisfy the **Security Configuration** requirements, the network is segmented as follows:

| Network Zone | Role | Exposed Ports | Accessible By |
| --- | --- | --- | --- |
| **Public** | Load Balancer | 80, 443 | Anyone (Internet) |
| **Private (Web)** | LiteSpeed Nodes | 80, 22 (Internal) | Load Balancer only |
| **Private (Data)** | Kafka, Redis, DB | 9092, 6379, 5432 | Web Tier & Consumers only |

## 5. Performance and Scaling Logic

* 
**Horizontal Scaling:** When traffic increases, additional LiteSpeed nodes can be added to the Load Balancer's backend pool without downtime.


* 
**Vertical Scaling:** For data-intensive tasks, the Kafka Broker and Redis nodes are assigned higher RAM/CPU allocations to handle larger message buffers and cache sizes.
