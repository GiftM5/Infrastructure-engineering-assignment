# Architecture & Network Flow Explanation

## Overview
This document provides a comprehensive overview of the architecture and network flow for the CashIT assignment. It outlines the various components involved in the system and how they interact with each other.

## Components
1. **Load Balancer**: Distributes incoming traffic across multiple web servers to ensure high availability and reliability.
2. **Web Servers**: Handle client requests and serve web pages. They communicate with the database and other services as needed.
3. **Kafka**: A distributed streaming platform that handles real-time data feeds. It acts as a message broker between producers and consumers.
4. **Redis**: An in-memory data structure store used for caching and session management.
5. **PostgreSQL**: A relational database management system that stores application data.
6. **HAProxy**: A high availability load balancer that manages traffic to the web servers.

## Network Flow
1. **Client Request**: A client sends a request to the load balancer.
2. **Load Balancer**: The load balancer routes the request to one of the available web servers based on the current load.
3. **Web Server**: The web server processes the request, which may involve querying the database or interacting with Kafka.
4. **Database Interaction**: If data is needed, the web server queries PostgreSQL and retrieves the necessary information.
5. **Caching**: Frequently accessed data may be cached in Redis to improve performance.
6. **Response**: The web server sends the response back to the load balancer, which then forwards it to the client.

## Detailed Flow Explanation

### 1. Client Request
- **Initiation**: The flow begins when a client (user) initiates a request, typically through a web browser or application interface. This request could be for accessing a webpage, submitting a form, or retrieving data.
- **Target**: The request is directed to the **[Load Balancer](01-load-balancer.md)**, which serves as the entry point for all incoming traffic.

### 2. Load Balancer
- **Traffic Distribution**: The load balancer's primary role is to distribute incoming requests evenly across multiple **[Web Servers](02-web-producer.md)**. This ensures that no single server becomes overwhelmed with traffic, which could lead to slow response times or downtime.
- **Health Checks**: The load balancer continuously monitors the health of the web servers. If a server becomes unresponsive, the load balancer will redirect traffic to the remaining healthy servers.

### 3. Web Server
- **Request Processing**: Once the load balancer routes the request to a specific web server, the server processes the request. This may involve:
  - **Rendering HTML**: Generating the webpage content to be sent back to the client.
  - **Business Logic**: Executing any necessary business logic, such as validating user input or processing transactions.

### 4. Database Interaction
- **Data Retrieval**: If the request requires data (e.g., user information, product details), the web server will query the **[PostgreSQL](06-db-writer.md)** database. This involves:
  - **SQL Queries**: Executing SQL commands to retrieve or manipulate data stored in the database.
  - **Data Integrity**: Ensuring that the data retrieved is accurate and up-to-date.

### 5. Caching with Redis
- **Performance Optimization**: To enhance performance, frequently accessed data may be cached in **[Redis](05-redis-server.md)**. This means:
  - **In-Memory Storage**: Redis stores data in memory, allowing for faster access compared to querying the database each time.
  - **Session Management**: Redis can also manage user sessions, storing session data temporarily for quick retrieval.

### 6. Kafka Messaging
- **Asynchronous Processing**: For operations that require real-time data processing (e.g., logging user actions, processing transactions), the web server may send messages to **[Kafka](04-kafka-broker.md)**. This involves:
  - **Producers**: The web server acts as a producer, sending messages to Kafka topics.
  - **Consumers**: Other services (consumers) can subscribe to these topics to process the data asynchronously, ensuring that the web server remains responsive.

### 7. Response to Client
- **Final Response**: After processing the request and gathering all necessary data, the web server sends the response back to the load balancer.
- **Load Balancer to Client**: The load balancer then forwards the response to the original client, completing the request cycle.

### Conclusion
This flow ensures that the CashIT application operates efficiently, handling multiple requests simultaneously while maintaining high availability and performance. Each component plays a vital role in ensuring a seamless user experience, from the initial request to the final response.
